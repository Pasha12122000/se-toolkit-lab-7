This bot will be built in four small increments so each task leaves the project in a working state. First, the scaffold creates a clean package layout with an entry point, configuration loader, handler layer, and service layer. The key architectural rule is that handlers must stay independent from Telegram-specific types, so the same command logic can be reused from CLI test mode and from the Telegram transport. That makes debugging faster and matches the autochecker workflow.

Second, backend integration will replace placeholder responses with real LMS API calls. A dedicated API client in `bot/services/` will handle authentication, request formatting, and friendly error messages so handlers remain simple. Third, natural-language routing will add an LLM client plus tool definitions for backend endpoints. The routing loop will let the model choose tools, receive tool results, and then produce the final user-facing answer. No regex-based intent shortcuts will be used in the routing path because the lab explicitly forbids replacing LLM routing with manual heuristics.

Finally, deployment work will package the bot into Docker and add a `bot` service to `docker-compose.yml`. At that stage we will switch container networking to service names instead of localhost, wire environment variables from secret files, and document deployment and verification in the README. After each task, the VM verification commands and Telegram checks should still pass.
